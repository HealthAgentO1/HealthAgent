## Issue 1 [P0] [security] SymptomSessionViewSet exposes all users‚Äô sessions (IDOR)

**Title:** Fix IDOR on `/api/symptom-sessions/` ‚Äî scope queryset and assign user on create

**Body:**
## Summary
`SymptomSessionViewSet` in `api/views.py` returns the full `SymptomSession` queryset for any authenticated user and calls `serializer.save()` without setting `user=request.user`. Any logged-in client can list, retrieve, update, delete, and `book` other users‚Äô sessions by primary key.

## Expected
- `get_queryset()` filters `SymptomSession.objects.filter(user=self.request.user)`.
- `perform_create` / updates enforce ownership (same pattern as `UserSymptomSessionsListView` / `UserSymptomSessionRetrieveView`).
- Explicit `permission_classes = [IsAuthenticated]` on the ViewSet (remove ‚Äútemporary‚Äù bypass comments).
- Regression tests: user A cannot GET/POST book on user B‚Äôs session.

## References
- `api/views.py` ‚Äî class `SymptomSessionViewSet` (comments say ‚ÄúTemporarily disabled for testing‚Äù)
- `api/tests/` ‚Äî add `test_symptom_session_viewset_scoping.py` (or extend existing session tests)

## Issue 4 [P2] Harden survey LLM endpoint (errors, payload size, prompt trust)

**Title:** Symptom survey LLM: distinct error types, tighter limits, production prompt strategy

**Body:**
## Problems
1. `SymptomSurveyLlmView` maps generic `Exception` to 502 ‚Äî hard to distinguish bad JSON vs network.
2. `system_prompt` max length 200k and arbitrary `user_payload` enable cost/DoS for authenticated users.
3. Production should not rely on client-supplied system text without allowlisting or server templates.

## Suggestions
- Map parse/upstream errors similarly to `SymptomChatView` where applicable.
- Lower caps; rate limit per user; optional server-side template keyed by `phase`.
- Log structured fields (user id, phase) without raw PHI where possible.

## References
- `api/views_symptom.py` ‚Äî `SymptomSurveyLlmView`
- `api/services/symptom_llm.py` ‚Äî `complete_symptom_survey_turn`
